# TokenSquash v0.3.0 Plan

Status: development planning after the published v0.2.2 release.

v0.3.0 should make TokenSquash harder to fool with small samples. The goal is
not to add a new compression claim. The goal is to prove whether deterministic
`ts1` / `tr1` evidence, real redacted corpora, and experimental sidecar reviews
can produce a trustworthy trend over time.

## Product Goal

TokenSquash v0.3.0 should answer this practical question:

> As the local redacted corpus grows, do token savings, meaning preservation,
> privacy checks, and release evidence get stronger, weaker, or stay noisy?

The deterministic codec remains the source of truth. Local-AI sidecar output is
still experimental and must be reviewed before it supports any public claim.

## Main Work

### Real Corpus Growth Track

Grow private real-turn evidence in staged milestones:

| Turns | Milestone | Required evidence |
|---:|---|---|
| `10` | `first_read` | scorecard, report, suggestions, certification |
| `25` | `pattern_probe` | scorecard-pack, alias-impact review, claim-pack draft |
| `50` | `trend_probe` | scorecard history, certification comparison, sidecar review sample |
| `100` | `benchmark_candidate` | release-check, claim-pack, public-safe summary |

Recommended local workflow:

```powershell
python -m tokensquash turns capture --prompt-file private-turns\prompt.txt --reply-file private-turns\reply.txt --evaluate --json
python -m tokensquash turns scorecard private-turns\real.redacted-turns.jsonl --counter chars --json --out private-turns\scorecards\current.json
python -m tokensquash turns scorecard-pack private-turns\real.redacted-turns.jsonl --counter chars --out-dir private-turns\scorecard-pack
python -m tokensquash turns certify private-turns\real.redacted-turns.jsonl --counter chars --out-dir private-turns\certification
python -m tokensquash turns claim-pack private-turns\certification --out-dir private-turns\claim-pack
```

Each milestone should keep raw turns local under ignored storage and commit only
redacted examples or aggregate public-safe evidence when needed.

### Evidence Trend Reports

v0.3.0 should make scorecard history useful as a development compass, not just
a release artifact. The preferred evidence should compare:

- current real-corpus scorecard against the previous local milestone
- current scorecard against the last published release evidence
- certification history for pass/watch/fail drift
- alias-impact and repeated-pattern changes as the corpus grows
- privacy finding count and raw wire loss count over time

Existing commands should stay useful:

```powershell
python -m tokensquash turns compare-scorecards private-turns\scorecards\previous.json private-turns\scorecards\current.json
python -m tokensquash turns scorecard-history private-turns\scorecards\10.json private-turns\scorecards\25.json private-turns\scorecards\50.json
python -m tokensquash turns compare-certifications private-turns\certifications\previous private-turns\certifications\current
python -m tokensquash turns certification-history private-turns\certifications\10 private-turns\certifications\25 private-turns\certifications\50
```

If the current comparison reports are too verbose for release review, add a
small trend summary layer rather than changing the underlying evidence schemas.

### Sidecar Meaning Evaluation

The sidecar should continue to be treated as a proposal layer. v0.3.0 should
focus on whether sidecar semantics survive round-trip review:

- `sidecar roundtrip` should remain inspectable for single examples
- `sidecar evaluate` should produce corpus-level savings and warning counts
- `sidecar review` should expose pass/watch/fail meaning-risk decisions
- `sidecar gate` and `sidecar certify` should make unsupported sidecar claims
  fail safely
- claim packs should separate deterministic claims from sidecar claims

Sidecar evidence can improve the product only when decoded text, warnings,
review rows, and certification status are all visible.

### Public Claim Guardrails

`turns claim-pack` should default to cautious language. Stronger public wording
should require:

- a named corpus size and counter
- privacy finding count of zero, or explicit documented handling
- adaptive benchmark behavior recorded
- scorecard or certification status of `pass`
- no unreviewed sidecar evidence used as canonical support
- clear limits explaining that results are corpus-specific

Unsupported wording should stay out of generated `claim.txt`, especially broad
claims such as universal token savings, production savings, or guaranteed model
behavior.

### Public Release Verification

v0.3.0 should make it simple for an outside reviewer to verify the latest
GitHub Release assets without needing local private evidence. The
`verify-github-release` command is the first wrapper for this path and covers:

- `gh release view`
- `gh release download`
- `verify-release-assets` against downloaded files
- installing the downloaded wheel in a temporary environment
- `about --json` and `demo --counter chars --json` smoke checks

The existing release verification guide remains the source of tracked expected
hashes and manual inspection detail.

## Acceptance Checklist

- README links to this v0.3.0 plan from the planning records.
- Product manifest and strict doctor include this plan in governance documents.
- `python -m tokensquash doctor --strict` passes.
- `python -m unittest discover -s tests` passes.
- The 10/25/50/100 real-corpus milestone workflow is documented with commands.
- Scorecard history or certification history can show trend direction across at
  least three saved evidence points.
- Claim-pack output remains conservative when corpus size is small or status is
  `watch`.
- Sidecar review/certification evidence keeps deterministic and sidecar claims
  visibly separate.
- Public release verification can be repeated from downloaded GitHub Release
  assets without access to private raw turns.
- No raw private prompts, replies, aliases, or local model output are committed.

## Out Of Scope

- Replacing `ts1` or `tr1`.
- Claiming universal token savings.
- Treating local-AI sidecar output as canonical.
- Adding hosted services or cloud-model dependencies.
- Publishing private raw turns or unreviewed sidecar outputs.
- PyPI publishing unless release verification is already boring and repeatable.
